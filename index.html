Babu Learns Gujarati
Gujarati Language App for Kids · ElevenLabs · GitHub Pages
The Problem

Over 1.5 million Gujarati-Americans live in the US. Most diaspora children hear Gujarati at home but have no engaging, age-appropriate tool to learn it actively. Existing apps like Duolingo weren't built for Indic languages or pre-readers. Parents want their kids to connect with their heritage — but there was no Babu for Gujarati.

What I Built

A mobile-first, single-file web app designed for children who cannot yet read. Babu (a cartoon monkey) speaks every word aloud via a custom ElevenLabs voice. Children swipe right for "Got it" and left for "Again" on picture-only flashcards. The quiz shows a 2×2 emoji grid — Babu says a word, the child taps the matching picture. No text, no buttons, no friction.

Key Product Decisions
  • Removed all on-card text — the card is just a picture. Babu says everything. Dramatically reduced cognitive load for pre-readers.
  • Swipe gesture replaces Got it / Again buttons — instinctive for touchscreen-native children.
  • Stars = words learned (1 per word, no quality score). localStorage flag prevents duplicate stars on replay.
  • Quiz distractors sourced from the same stack only — a fruits quiz never shows a color or family emoji.
  • Two-strike wrong answer rule: encourage on first miss, reveal the correct answer on second — prevents children from getting stuck.
  • Babu animates: jumps on correct, flips on wrong — replaces text feedback entirely.
  • Gujarati praise appears first on the reward screen, English below — reinforces Gujarati as the primary language.
  • ElevenLabs eleven_multilingual_v2 chosen over Sarvam for V1 — faster to market. Sarvam upgrade planned for V2 (authentic Gujarati phonemes and STT pronunciation feedback).
Tech Stack
ElevenLabs TTS eleven_multilingual_v2 Vanilla JS GitHub Pages localStorage Web Speech API (fallback) CSS Swipe Gestures
What's Next
  • V2: Pronunciation feedback via Sarvam STT, 3 new vocabulary topics, pre-generated MP3s so no API key ships to the browser
  • V2.5: iOS App Store via Capacitor wrapper
  • V3: Supabase backend, parent dashboard, cross-device progress sync

USPS Chatbot V1
RAG Agent · Claude · Supabase pgvector
The Problem

USPS customers frequently complain that it is hard to find the answer to their questions. The information exists in USPS FAQ documentation, but customers cannot navigate to the correct page. The opportunity: an AI agent that answers accurately from trusted content and knows when to escalate.

What I Built

A RAG-powered conversational chatbot that scrapes and indexes 286 real USPS FAQ articles, retrieves the most semantically relevant content for each question, and uses Claude to generate a grounded, plain-language response. The agent escalates gracefully when confidence is low — routing users to a real human rather than hallucinating an answer.

Key Product Decisions
  • Used Firecrawl over BeautifulSoup — USPS FAQs are JavaScript-rendered via Salesforce Experience Cloud, which returns empty pages to traditional scrapers
  • Switched from sentence-transformers to Voyage AI (voyage-3) after evals showed sentence-transformers failed on the most common USPS queries — Hit Rate@5 jumped from ~80% to 100%
  • Set similarity threshold at 0.3 after running a 20-question golden set (10 common + 5 niche + 5 escalation) — LLM-as-Judge scored 4.93/5 faithfulness, 4.73/5 relevance
  • Chose Supabase pgvector over Pinecone to reduce infrastructure complexity and stay free-tier compatible for a V1
  • Deployed on Render free tier with a cron-job.org keep-alive ping every 10 minutes to prevent cold starts
Learning Objectives & Summary

I wanted to vibe-code a conversational AI chatbot end-to-end — with hands-on exposure to tools like Firecrawl, Supabase, Claude Code, and Git — while thinking through the full product lifecycle: data acquisition, embedding strategy, retrieval quality, agent design, hosting tradeoffs, cost constraints, and frontend UX. The goal was to develop sharp instincts for where AI adds value, where it breaks, and what decisions a PM needs to make at every layer of the stack.

Tech Stack
Claude Sonnet 4 Voyage AI (voyage-3) Firecrawl Supabase pgvector FastAPI Render GitHub Pages Python cron-job.org
Thoughts for V2
  • Streaming responses for a more natural, real-time chat feel
  • Conversation memory so follow-up questions work naturally across turns
  • Summarize retrieved articles before generating a response — improving answer quality and reducing token usage
  • Clarifying questions — when a query is ambiguous, ask one follow-up before retrieving to surface a more precise answer

Easy Decision Maker
AI-Powered Decision Intelligence
The Problem

People face complex decisions every day — career moves, business strategies, major purchases — but lack a structured way to think them through. Generic pros/cons lists are shallow. Consultants are expensive. Most people just wing it.

What I Built

Easy Decision Maker uses Google Gemini to automatically transform vague prompts into structured, professional-grade analytical frameworks — including Comparison Tables, SWOT Analysis, and PESTLE Analysis. The AI selects the right framework based on your input, and a Clarification Gate pauses generation when prompts are too vague to ensure high-quality outputs.

Key Product Decisions
  • Retired lower-value frameworks (SOAR, Porter's 5 Forces) to reduce cognitive overload
  • Added server-side rate limiting (10 req/day) to control API costs and ensure sustainability
  • Designed mobile-first with auto-expanding inputs for on-the-go decision making
  • Built strict JSON schema validation to guarantee 100% structured output integrity
Learning Objective & Summary

My learning goal with this page was to use Google AI studio and vibe code an app. The AI created many errors with button usage and wasn't inherently mobile-friendly, showing me where to focus more of my time in Google AI Studio.

AI PM Portfolio Website
Configurable Single-Page Portfolio · GitHub Pages
The Problem

Most PM portfolios are static PDFs or generic LinkedIn profiles — they don't show how a product manager thinks, builds, or ships. As someone pivoting into AI PM roles, I needed a living showcase that could host real, interactive demos and be updated quickly without a full engineering setup.

What I Built

A single-file, zero-dependency portfolio site hosted on GitHub Pages under my own namespace (rujutagandhi.github.io). It features a scrollable layout with sections for About, Experience, Education, Skills, and AI Demos. Each demo has a card grid entry and a detailed modal with product context — problem, decisions, and tech stack. The entire site lives in one HTML file, making it easy to maintain and version-control.

Key Product Decisions
  • Single-file architecture — No build pipeline, no dependencies. Push one file to GitHub and it's live instantly. Reduces friction for continuous updates.
  • Modal-based demo detail — Instead of separate pages per demo (which would need routing), modals keep the user in context and load instantly with no extra network requests.
  • Demo-card pattern — Standardized Problem / What I Built / Key Decisions / Tech Stack structure signals PM thinking rigor, not just developer output.
  • GitHub Pages hosting — Free, SSL-secured, and tied to the professional namespace rujutagandhi.github.io. Zero ops overhead.
  • AI-assisted development with Claude — Used Claude as a co-pilot for design and code generation, demonstrating practical AI tool fluency to hiring managers.
Learning Objective & Summary

While working on AI Demo's, I will always have a learning goal. Through this website, I wanted to learn and demonstrate use of Github & Github CLI and Claude (Projects, Prompting). This AI Demo website was developed in combination with the two. I also leveraged previous HTML/CSS knowledge to make further edits without Claude use to be mindful of token usage.

Tech Stack
HTML5 CSS3 (custom properties) Vanilla JavaScript GitHub Pages Claude Fraunces + Outfit (Google Fonts)
